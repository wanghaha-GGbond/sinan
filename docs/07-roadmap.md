# 07 · 工程路线图 M0-M4

> Sprint 级任务拆解见 [DEV_PLAN.md](DEV_PLAN.md)（S0-S10 覆盖 M0-M2）。本文档给里程碑视角：
> 每阶段的工程交付、依赖关系、明确不做清单。节奏假设：2 周一个 Sprint，1-2 名全栈。

## 依赖关系（为什么是这个顺序）

```
验证闭环(L1/L2) ──> trustLevel ──┬──> 身份卡材质/段位标签
                                ├──> 邀请配额发放（L1 达成才发）
                                ├──> 违约库发布门槛（L2 起）
                                └──> k-匿名样本计数（按验证用户数）
departments ──> 部门级评分聚合 ──> 情绪指数 ──> 分享卡 ──> 增长漏斗
邀请制 ──> 种子 500 人 ──> 内容密度 ──> M2 引爆素材
M2 运营拍卖(零代码) ──> 真实竞价数据 ──> M3 拍卖系统设计
```

**一切等级逻辑的开关是「验证 approved → trustLevel 提升」这一个事务**，它是 S1 的第一优先任务。

## M4 想象力（第 10-12 月）

- 积分竞猜内测（预测市场的合规包装，法务先行，封闭内测）
- 可视化公司（组织热力图 / 招聘动向，数据源 = 站内验证用户分布 + 公开招聘数据）
- B 端探索：雇主品牌体检报告（**只卖聚合洞察，不卖任何可定位个人的数据**，红线见 08 §3）

## 明确不做清单（防 scope creep，每个 Sprint 复读一遍）

- M3 前：拍卖竞价系统、L3、圈层、私聊、预测市场、可视化公司、高光馆、一技封神
- 全程（直到另行评审）：NLP 情感分析（评分代理够用）、BI 系统（SQL+admin 页够用）、移动 App（Web 优先）、开放注册（邀请制是定位不是阶段）
- 永远不做：出售个人数据、竞对定向爬取、任何形式的「付费删差评」（这一条写进公司章程级红线）

## 工程贯穿规则

1. 任何新展示面上线前过 `lib/server/anonymity.ts`，不允许绕过（08 §2）。
2. 违约库、身份卡敏感字段一律先审后发；普通评价维持现有状态机。
3. PIPL 最小化：schema review 时对每个敏感字段问一句「能不能不存」。
4. 本仓库 Next.js 版本有 breaking changes：写代码前读 `node_modules/next/dist/docs/` 对应章节（AGENTS.md 约定）。
