# 04 · F3 邀请制 + 圈层私聊 Spec

> 增长杠杆。原则：**邀请是声誉背书，不是拉新任务**。
> 代码现状：注册完全开放（`api/auth/register/route.ts`），无邀请概念；私聊、圈层均无。
> 上线节奏：邀请制 M0；圈层与私聊 M3（但 M0 的数据结构要为其留口）。

## 1. 邀请制（M0）

### 1.1 数据模型（新增）

```
invites
  id, code (唯一短码, 8位, 去易混字符),
  inviter_user_id (FK users), invited_user_id (FK users, nullable),
  status (unused / used / revoked),
  created_at, used_at
```

被邀请关系永久展示：`app/u/[id]` 与 `/me` 显示「由 XX 引荐」（链到邀请人主页）。
这条关系同时是追责链：被邀请人造假 / 作恶，邀请人声誉分连带受损（v1 只记录不扣分，M1 起生效连带规则）。

### 1.2 配额规则（`lib/server/invites.ts` 单一入口）

| 规则 | 值 | 理由 |
|---|---|---|
| 初始配额 | 用户达到 **trustLevel ≥ 1 时**发放 3 枚 | 注册即发会被小号自激活刷爆 |
| 返还机制 | 被邀请人达到 trustLevel ≥ 2 时，邀请人 +1 枚 | 见 1.4 的 K 因子数学 |
| 配额上限 | 6 枚 | 守住「终身稀缺」叙事 |
| 邀请码有效期 | 不过期，但可由邀请人撤回（revoked） | |

### 1.3 注册流改造

- `POST /api/auth/register` 增加 `inviteCode` 必填校验；注册成功**同一事务内**核销邀请码 + 写入 `invited_user_id`。
- 环境变量 `INVITE_REQUIRED`（默认 true）作总开关，内测期 / 种子导入期可关。
- `app/register/page.tsx`：邀请码输入框，支持 URL 预填 `?invite=`。
- **邀请落地页 `app/invite/[code]`（K 因子成败在这一页）**：被邀请人打开链接看到——邀请人的身份卡（材质即背书）+ 一句价值主张 + 「你被引荐进入」的仪式感。M2 前持续 AB 打磨。

### 1.4 K 因子数学（为什么需要返还机制）

```
K = 人均发出邀请数 × 被邀请人注册转化率
目标 K > 0.8；若使用率 50%：3 枚 × 50% = 1.5 发出/人 → 需注册转化 > 53%（紧绷）
加返还：活跃邀请人实际配额 4-6 枚 → 同等转化率下 K 上限 1.6-2.4（留出余量）
```

监控口径见 09；若 K 持续 <0.5，优先修落地页转化而不是加配额——**加配额会稀释背书价值，是最后手段**。

### 1.5 种子期运营（M0，500 人）

- 种子用户由运营定向邀约（只邀 Top 20 公司），走专用批量邀请码（`inviter_user_id` 指向官方账号，主页显示「创始会员」而非「由 XX 引荐」）。
- 验收：种子 500 人中 L1 转化 >60%（定向邀约人群应高于自然流量）。

## 2. 圈层（M3，本节为设计预案）

```
circles
  id, name, slug, description, min_trust_level, status, created_at
circle_members
  circle_id, user_id, endorsed_by_user_id (FK users), joined_at, status
```

- 首批圈层：总监圈 / 出海圈 / 大模型圈（运营建圈，不开放自建）。
- 入圈 = 1 名现有成员背书（`endorsed_by_user_id`）+ 验证等级门槛（`min_trust_level`，总监圈要求 L2+）。
- 背书与邀请同理是声誉连带：背书人可见于成员页。
- 圈内内容默认圈内可见；圈层徽标展示在身份卡（M3 实现）。

## 3. 私聊段位规则（M3，本节为设计预案）

- 同级或相邻段位：直接私聊。
- **跨两级及以上：进入「请求队列」**——接收方在「请求」箱按自己节奏处理，不产生未读红点、不推送。高段位用户的注意力是平台最稀缺资源，规则的本质是保护它。
- 请求附 140 字自我介绍（与拍卖「为什么是我」同一交互范式，见 05）。
- 数据模型：`dm_threads / dm_messages / dm_requests`，M3 细化；端到端加密不承诺（合规要求可审查），但承诺员工不可见明文（应用层加密 + 审计访问）。

## 4. 验收标准

| 指标 | 目标 |
|---|---|
| 邀请使用率（发出 / 配额） | >50% |
| K 因子 | >0.8 |
| 邀请落地页 → 注册转化 | >53%（K 目标的推导值，单列监控） |
| 被邀请人 L1 转化 | >60%（背书带来的信任溢价应体现在这里） |
| 圈层（M3） | 首批 3 圈各 ≥50 人，背书拒绝率留观 |
