# 08 · 合规与风险：名誉权 / PIPL / k-匿名 / 审核 SOP

> 本产品的三大风险（法律、真实性、冷启动）中，前两个的工程化应对全部落在本文档。
> 红线原则：**库里可以存真话，展示层必须可辩护。**

## 1. 名誉权（对公司）防线

公司名誉权诉讼是头号法律风险。防线分四层：

1. **结构优先**：评价主体是五维打分 + 结构化条目（事实陈述），自由文本全部过审核状态机（已有 `review_status` 管线）。打分是意见表达，受到的保护强于事实断言——**让产品形态本身成为抗辩理由**。
2. **事实条目可溯源**：违约库「承诺 vs 兑现」要求证据指纹（03 §2.3）；情绪指数事件标注只陈述事实不下判断（「传出裁员消息」✓ /「血洗式裁员」✗）。
3. **申诉通道畅通且留痕**：公司方走 `company_appeals`（已有表，reason 枚举含 fake_fact/rumor/competitor 等）。受理 SLA 5 个工作日；申诉成立 → 内容下线 + 写审计事件；**申诉成立率是守门指标**（09），它异常升高说明审核在放水。
4. **绝不卖删帖**：「付费删差评」是脉脉等前辈被锤的核心原因，也是社区信任的死穴。公司方唯一合法路径是申诉通道。写进章程级红线（07）。

## 2. k-匿名规则（保护发布者 = 保护平台）

部门颗粒度 × 段位标签 = 可反推身份。一个 30 人部门里 L7 只有两三个，「某 L7 验证用户」形同实名。
这同时是：发布者人身风险、平台法律风险（被诉时员工被反推）、供给抑制（用户不敢写）。

### 规则（唯一实现入口 `lib/server/anonymity.ts`）

| 条件 | 行为 |
|---|---|
| 部门内 trustLevel≥1 的发布者 ≥5 人 | 部门级展示 + 精确段位标签（「某 L7 验证用户」） |
| 部门内发布者 <5 人 | 评价归并到公司级展示；部门 tab 显示「样本不足」 |
| 公司内同段位发布者 <3 人 | 段位标签模糊化（「某 L7」→「某高 P」） |
| 违约库条目 | 不论样本量，**一律模糊段位**（内容本身已高度可定位） |

- 阈值（5 / 3）作为配置常量，可调不可绕过；任何新展示面（页面或 API）必须经过该模块，code review 检查项。
- 存储层保留全量真实数据，模糊只在出口（06 §5）——规则可随样本密度演进。
- 单元测试覆盖边界：恰好 5 人、跨状态（hidden/deleted）评价是否计入、用户吊销验证后的重算。

## 3. PIPL 合规

| 数据 | 策略 |
|---|---|
| 验证凭证原件（工牌/证明） | **审完即删**，文件存储 7 天 TTL；只留审核结论 + 指纹（02 §4） |
| 薪资信息（L3） | 敏感个人信息：单独同意 + PIA 先行，M3 前不碰（02 §7） |
| 违约证据原件 | 不入库，留 SHA-256 指纹（03 §2.3） |
| 企业邮箱 | 验证用途单独告知；防留痕发信设计（02 §3） |
| B 端报告（M4） | 只卖聚合洞察；任何输出满足本表 §2 同等的 k-匿名约束 |

通用动作：隐私政策按功能里程碑同步更新；新增敏感字段在 schema review 时强制回答「能不能不存」；用户注销走 `deleted_at` 软删 + 30 天硬删队列（待 M1 实现）。

## 4. 审核 SOP

### 4.1 验证审核（L2，48h SLA）

查验要点：工牌/证明与所选公司一致性、姓名一致性、日期有效性、图像 PS 痕迹（边缘/字体/光照）；存疑 → 要求补充第二凭证；通过后立即删除原件。
每笔审核记录审核人（`reviewed_by_user_id`），吊销率 >1% 触发审核质量复盘（02 §8）。

### 4.2 内容审核（评价/讨论）

沿用现有状态机（pending_review → visible/limited_visible/hidden/rejected）。优先队列：被举报内容 > 违约库 > 新用户首帖 > 存量。
`review_report_reason` 已含 competitor / company_astro（公司水军）——这两类举报单独看板，关联 §6 反作弊。

### 4.3 违约库审核（最严）

全人工先审后发。三查：承诺是否具体可证伪、兑现描述是否事实化、证据描述与指纹是否齐全。任何「情绪输出包装成事实」退回修改。

### 4.4 身份卡敏感字段（高光时刻 / 拒绝陈列位）

防三件事：冒充（声称不属实的经历）、拉踩（贬低前东家）、暴露第三方（写出可识别的他人）。

## 5. 审计要求

任何「内容从可见变为不可见」「身份等级变更」「申诉处理」必须可追溯到操作人和时间。
实现：`discussion_moderation_events` 模式推广（06 §4）。审计日志只增不删，与业务数据分离备份。

## 6. 真实性与反作弊（竞对互黑）

1. **门槛即防线**：发布须 L1+，违约库 L2+——伪造成本 = 伪造在职证明的成本。
2. **异常聚类检测**（M1 起，先报表后自动）：同公司短时负面密度突增、新账号集中评价单一竞对、同设备指纹多账号（`reviews.author_fingerprint_hash` 已预留）。
3. **邀请链追责**：被邀请人作恶，邀请人声誉连带（04 §1.1），让作恶污染社交资产。
4. 处置阶梯：limited_visible（已有状态）→ 隐藏 + 通知 → 吊销验证（revoked）→ 封号。每级写审计。

## 7. 内容安全（平台自身合规）

- 互联网信息服务算法备案 / 安全评估：情绪指数上线前评估是否触发「舆论属性」申报义务——**这是 M1 的法务作业，不是 M4 的**。
- 敏感词与涉政过滤：接成熟内容安全 API（自建不现实），在 pending_review 前置。
- ICP：非经营性备案先行；M3 变现前升级经营性评估（05 §5）。
